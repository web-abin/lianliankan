# 微信小游戏设计文档

## 1. 文档目的

- 本文档基于最新的 `docs/需求文档.md` 输出，聚焦“页面布局设计”而非玩法规则本身。
- 目标是为首页、加载页、局内页、商店、主题、排行榜以及各类弹窗提供统一的布局基线、信息层级和视觉结构说明。
- 文档优先服务微信小游戏竖屏场景，要求兼顾安全区、胶囊按钮、全面屏底部手势区、弱网环境下的可读性与点击可靠性。

## 2. 设计目标

- 整体气质：治愈、轻松、软糯、有呼吸感，避免过强竞技压迫感。
- 信息原则：关键内容一眼可见，非关键内容弱化展示，不让 UI 抢走局内注意力。
- 交互原则：大按钮、大留白、少层级，所有核心操作都要适配手指点击。
- 布局原则：全屏页优先沉浸感，局内页优先信息效率，弹窗优先决策明确。
- 风格原则：统一圆角、柔和投影、浅暖配色、卡皮巴拉主题插画承接情绪。

## 3. 全局布局规范

### 3.1 设计基线

- 统一以 `750` 为设计参考宽度。
- 全屏页面使用“按屏幕比例推导高度”的逻辑高 `designLayoutH`，保证首页、加载页、排行榜等页面在超长屏上不出黑边。
- 局内页与弹窗仍按 `750 x 1334` 的设计稿体系实现，确保棋盘和弹窗结构稳定。
- 需要特别区分：`750 x 1334` 仅用于布局参考，不等于背景图原始出图尺寸。
- 对于首页、加载页、商店、主题、排行榜这类全屏页面，背景图素材建议使用“更高一档”的出图尺寸，避免长屏设备上下留白。

### 3.1.1 背景图出图规范

- 全屏背景图不要只按 `750 x 1334` 出图。
- 推荐背景图原始设计尺寸至少为 `750 x 1750`。
- 背景图的核心主体内容应尽量落在中部安全区，顶部和底部预留可裁切的延展区域。
- 顶部延展区域适合放天空、云层、树冠、光斑等弱信息元素。
- 底部延展区域适合放草地、桌面、地板、浅纹理、阴影等弱信息元素。
- 严禁把标题、角色脸部、主按钮承托光效、关键建筑边缘等重要内容贴在背景图上下边缘。
- 对于局内页背景，如果棋盘上下仍可能露出较多背景，也建议使用高于 `1334` 的背景源图，再按布局裁切显示。

### 3.2 安全区规范

- 顶部安全线需同时避开状态栏与微信右上角胶囊按钮。
- 顶部核心信息区的起始 Y 必须低于安全区底边，并额外预留一档呼吸间距。
- 右上角功能按钮、菜单按钮、关闭按钮不得进入胶囊按钮左侧危险区。
- 底部主操作按钮需高于底部手势安全线，避免误触系统返回区域。

### 3.3 页面分层规范

- `背景层`：大图背景、氛围光斑、纯色兜底。
- `主体层`：角色、棋盘、列表、主卡片。
- `信息层`：顶部状态、标题、页签、数值、进度。
- `交互层`：按钮、页签、工具位、关闭按钮。
- `浮层`：蒙层、机制说明、弹窗、奖励提示、排行榜开放数据域画布。

### 3.4 通用间距与尺寸建议

- 页面左右安全边距：建议不小于 `24` 设计单位。
- 主卡片圆角：建议 `24~32`。
- 次级卡片圆角：建议 `16~24`。
- 主要按钮高度：建议 `88~104`。
- 次级按钮高度：建议 `72~88`。
- 图标型按钮可视觉较小，但实际点击区不小于 `88 x 88` 逻辑像素。

### 3.5 通用信息样式

- 所有叠加在复杂背景上的文字，都需要描边、阴影或浅色底板提升可读性。
- 数值型信息优先“图标 + 数字 + 小底板”组合，避免纯文字裸露在背景图上。
- 红、绿、橙等状态色不单独表达含义，必须配合图标、标签或文案。

### 3.6 动效节奏原则

- 页面进入动效以淡入、轻微上浮、轻弹为主，不使用大幅横移。
- 局内 HUD 动效保持克制，避免遮挡棋盘或引发视觉疲劳。
- 弹窗动效建议控制在 `200~300ms` 内，并支持未来的“降低动态效果”设置扩展。

## 5. 加载页

### 5.1 页面定位

- 承接游戏启动后的第一印象。
- 用温和、稳定的视觉节奏消化资源加载等待感。

### 5.2 布局结构

- `背景层`
  - 全屏背景图改为“日式露天温泉泡澡”场景。
  - 场景主体建议包含木质温泉池、石头围边、远处树影或竹林、暖色灯笼/木屋一角，以及清晨或傍晚的柔和天光。
  - 温泉池中可放 1 至 2 只陪伴小动物，建议选择与卡皮巴拉气质一致的温和动物，如小鸭、白团雀、小乌龟或趴在池边的小猫，但不要喧宾夺主。
  - 背景图需宽度铺满、高度等比裁切，并在底层铺一层接近主色的纯色兜底。
- `顶部留白区`
  - 不放核心信息，保留呼吸感，避免启动即显得拥挤。
  - 可利用上方留白放天空、树枝、蒸汽和微弱光斑，作为长屏延展区域。
- `中部主视觉区`
  - 页面视觉中心放置泡在露天温泉中的卡皮巴拉。
  - 角色以“露出头部和上半身、表情放松、轻微飘浮在温泉里”的状态呈现，强化治愈和松弛感。
  - 可搭配一条简短加载文案，如“正在准备今天的温泉时光”。
- `中下进度区`
  - 进度条横向居中，宽度建议占屏宽 `60%~72%`。
  - 进度条整体不做传统硬条，而改为“温泉热气慢慢冒满”的表现。
  - 可将进度容器设计成一条扁圆形温泉池口或木质温泉铭牌，内部用半透明白色蒸汽、暖雾和细小气泡逐步填充。
  - 进度提升时，热气从左向右逐渐变浓，局部伴随轻微上升漂浮感，形成“温泉越来越暖、雾气越来越足”的节奏。
  - 进度条上方显示 `Loading...` 或简短中文文案。
- `底部提示区`
  - 固定展示健康游戏忠告。
  - 文案块整体贴底，但需高于底部手势安全线。

### 5.3 视觉层级

- 温泉中的卡皮巴拉角色 > 温泉热气进度条 > loading 文案 > 健康提示文案。
- 健康提示文案为最低视觉优先级，使用较轻的对比度，但必须清晰可读。

### 5.4 实现建议

- 角色插画与进度条之间保留明显纵向空隙，防止拥堵。
- 温泉热气进度条的填充动画应偏慢、偏柔和，避免像普通下载条那样机械推进。
- 蒸汽表现建议采用“低对比半透明白雾 + 少量暖色高光”，不要做成浓烟。
- 进度百分比可不单独显示数字，优先保留简洁感。

## 6. 首页

### 6.1 页面定位

- 首页是玩家的长期停留页，也是资源、入口、主线推进信息的核心枢纽。
- 需要同时满足“舒服”、“清楚”、“愿意点”的三重目标。

### 6.2 布局结构

- `背景层`
  - 首页背景为固定场景，不随主题切换而变化。
  - 场景参考当前示意图方向，采用清新、明亮、手绘感的草地治愈场景。
  - 背景主体建议为浅蓝天空、远景树影、淡绿色草坡、蒲公英、小花、零星小动物等元素，整体氛围轻松、松弛、儿童绘本感更强。
  - 背景主体尽量集中在中部偏下，顶部为状态区预留干净天空区域，便于放置状态信息与避让胶囊按钮。
- `顶部状态区`
  - 位于左上安全区内，以横向排布为主。
  - 包含设置按钮、金币显示、血量显示。
  - 设置按钮最靠左，金币与血量依次向右展开。
  - 整个区域建议使用半透明浅底板或分离式胶囊小卡片，避免文字压在复杂背景上。
  - 设置按钮需要以“小插画图标”风格设计，具有直观的齿轮形象，并带黑色描边，风格与金币、血量的插画小卡一致。
  - 设置按钮、金币、血量胶囊需要带清晰的黑色描边，保证在浅色天空背景中也有稳定识别度。
- `中央展示区`
  - 页面视觉锚点，放标题与卡皮巴拉待机动画。
  - 标题在上，角色在下，二者共同组成首页中心。
  - 卡皮巴拉角色尺寸需足够大，成为“想打开游戏看它动一下”的情绪锚点。
- `左右功能入口区`
  - 左右两侧各一列竖向按钮组，围绕中央角色分布。
  - 左侧：圈子好礼、添加桌面、商店、主题。
  - 右侧：喊人、每日奖励、排行榜、每日挑战。
  - 每个入口采用“独立图标 + 底部贴边文字”形式，不使用圆形按钮，也不使用统一底板背景。
  - 每个入口图标直接表现功能含义，例如奖杯、木牌、日历、电话、星星、画板等，让每个入口都像一个独立的小插画图标。
  - 图标本身需要有清晰的外轮廓线条，并统一带黑色描边，边缘明确，便于在浅色草地背景上被快速识别。
  - 文字不要完全放在图标下方另起一行，而是贴在图标组件底部，形成类似 `position:absolute; bottom: 0` 的视觉效果。
  - 入口文字同样需要带黑色描边，保证在草地、小花和浅色背景上依然清楚可读。
  - 左右两列入口的文字底边应统一对齐，保证整体节奏干净，不出现有的标签过高、有的标签过低的问题。
- `底部主行动区`
  - 放置“开始游戏”大按钮。
  - 主按钮上方或按钮内部展示当前主线关卡，如“主线 第 12 关”。
  - 主按钮必须是全页最强视觉 CTA。
  - “开始游戏”按钮文字需要带明显的黑色描边，保证在暖橙色按钮底和手绘纹理上依然醒目。

### 6.3 信息层级

- 第一优先级：开始游戏。
- 第二优先级：金币、血量、当前关卡。
- 第三优先级：每日挑战、每日奖励、商店、排行榜等外围功能。
- 第四优先级：标题与装饰性元素。

### 6.4 布局细节

- 中央角色区域不被按钮侵占，保证角色完整露出。
- 左右功能入口应形成近似对称的视觉秩序，但不必完全机械对齐，可保留少量高低错落，增加生动感。
- 左右功能入口更接近“漂浮在场景里的功能徽记”，而不是一组统一按钮组件，因此图标风格要统一，但轮廓造型可以各不相同。
- 所有入口标签都应以底部贴边方式排版，标签的底边线保持一致，避免图标大小不同导致文字参差不齐。
- 顶部状态区与中央标题区之间留足垂直间距，避免屏幕上半部分过满。
- 底部主按钮宽度建议在屏宽 `56%~68%` 之间，不能过窄。

### 6.5 状态变化

- 血量不足时，开始游戏按钮变为弱化态，但仍保留可点击入口，用于触发提示。
- 新奖励可领取时，对应入口挂红点或轻微呼吸光效。
- 首页返回后若发生奖励到账，金币/血量数字做轻量跳动反馈。

## 7. 首页设置弹窗

### 7.1 页面定位

- 首页的通用系统设置面板。
- 需要轻量、清楚、易关闭，不应做成过于厚重的功能页。

### 7.2 布局结构

- `蒙层`
  - 深色半透明，压低背景存在感。
- `主弹窗`
  - 居中卡片，建议纵向结构。
  - 顶部为标题区，中部为设置项列表，底部为问题反馈入口与关闭按钮。
- `标题区`
  - 中间显示“设置”。
  - 右上角放关闭按钮，但仍需避开胶囊按钮危险区；若弹窗本身已经居中，可将关闭按钮放在弹窗内右上角。
- `设置项列表`
  - 三行开关项：音效、震动、音乐。
  - 每行采用“左侧图标+标签 + 右侧开关”布局。
  - 行高要足够大，保证点击舒适。
- `底部操作区`
  - 放置“问题反馈”按钮。
  - 若能力不可用，展示置灰态或弱化提示。

### 7.3 设计建议

- 开关项建议做成柔软圆角条目，选中时使用主题色高亮。
- 列表项之间保持统一分隔节奏，不使用硬线分隔，优先间距和浅底色分组。

## 8. 圈子好礼弹窗

### 8.1 页面定位

- 这是首页功能入口的奖励弹窗，目标是引导点击微信原生游戏圈按钮。

### 8.2 布局结构

- `顶部标题区`
  - 标题：“圈子好礼”。
  - 可加一行副标题，说明加入游戏圈可领取奖励。
- `中部奖励展示区`
  - 大图标展示 `100 金币` 奖励。
  - 奖励数值要成为视觉中心。
- `说明区`
  - 简短说明“完成跳转后返回游戏即可领取”。
- `主操作区`
  - 预留微信原生按钮区域。
  - 原生按钮应放在弹窗主按钮位置，周围留出安全点击空间。
- `关闭区`
  - 次级关闭按钮置于底部或右上角。

### 8.3 设计重点

- 原生按钮位置必须稳定，不能被装饰元素包围得太紧。
- 文案要弱化技术实现感，突出“轻松领一份好礼”的体验。

## 9. 添加桌面弹窗

### 9.1 页面定位

- 用来承接“添加到手机桌面”的引导与每日奖励说明。

### 9.2 布局结构

- `头部`
  - 标题：“添加桌面”。
  - 副标题说明“添加后每日可领奖励”。
- `中部示意区`
  - 用一张简洁示意插图表现“小游戏图标进入桌面”的过程。
- `奖励区`
  - 强调“今日首次奖励 50 金币”。
- `底部按钮区`
  - 主按钮：“去添加”。
  - 次按钮：“稍后再说”。

### 9.3 设计重点

- 这是引导型弹窗，布局上需要“示意图 + 奖励 + 主按钮”顺序清晰。
- 主按钮和次按钮要有明显主次，不可做成等权。

## 10. 每日奖励弹窗

### 10.1 页面定位

- 承接连续签到，重点突出“今天能领什么”和“已经连续几天”。

### 10.2 布局结构

- `标题区`
  - 标题：“每日奖励”或“今日签到”。
  - 次级信息展示连续签到天数。
- `签到日历区`
  - 中部为 `7` 格奖励卡片。
  - 每格展示第几天与对应金币数。
  - 已领取、今日可领、未来未解锁需明显区分。
- `奖励说明区`
  - 今日奖励可在日历下方放大展示。
- `底部操作区`
  - 今日可领取时显示主按钮“领取奖励”。
  - 今日已领取时显示完成态按钮或说明文案。

### 10.3 设计重点

- 今日卡片需要明显高亮，建议使用放大、描边、发光中的至少两种。
- 超过 7 天后沿用第 7 天奖励，因此第 7 格应具备“长期收益”意味，可加循环标识。

## 11. 商店页

### 11.1 页面定位

- 商店承担金币消耗和养成购买，是功能页而非轻弹窗。
- 页面信息较多，需要明确标签切换结构。

### 11.2 页面总体结构

- `背景层`
  - 延续主题背景，但亮度和细节应弱于首页，避免干扰商品阅读。
- `顶部栏`
  - 左侧返回按钮。
  - 中间标题“商店”。
  - 右侧金币余额展示。
- `页签区`
  - 顶部栏下方设置三枚页签：形象、道具、音效。
  - 页签建议做成横向胶囊分段，选中态明确。
- `内容区`
  - 主体为商品卡片列表。
  - 视商品数量采用两列卡片或单列宽卡结构。
- `底部留白区`
  - 仅做安全留白，不额外堆叠固定操作栏。

### 11.3 形象页布局

- 商品适合采用“大卡片纵向列表”。
- 每张卡片包含：预览图、名称、价格、状态标签、购买按钮。
- 已拥有商品显示“已拥有”标签；未开放商品显示“敬请期待”。
- 当前使用中的形象应有明确标识。

### 11.4 道具页布局

- 道具与血量适合采用“两列网格卡片”。
- 单张卡片包含：图标、名称、功能一句话、价格、购买按钮。
- 道具属于高频小额购买，按钮要直观，减少说明文字占比。

### 11.5 音效页布局

- 使用“列表卡片 + 试听状态位”的预留布局更合理。
- 当前版本即使没有完整试听能力，也建议保留音效包的预览区结构。
- 卡片结构：封面图标、名称、风格描述、价格、购买/已拥有状态。

### 11.6 设计重点

- 顶部金币余额必须常驻可见，减少用户来回确认“够不够买”。
- 商品卡片不宜过小，避免一屏塞太多造成廉价商城感。
- 购买按钮位置全页统一，强化扫描效率。

## 12. 主题页

### 12.1 页面定位

- 主题页本质上是“收藏陈列页 + 切换页”。
- 需要同时承担展示美术差异、说明解锁条件、提供切换反馈。

### 12.2 页面总体结构

- `顶部栏`
  - 左侧返回。
  - 中间标题“主题”。
- `上半预览区`
  - 展示当前选中主题的大幅背景预览。
  - 预览区可叠加 2~4 个主题图块样例，帮助玩家快速理解风格。
- `中部信息区`
  - 展示主题名、主题一句话说明、解锁条件或已解锁状态。
- `下半列表区`
  - 主题卡片采用横向分页或纵向列表均可。
  - 推荐使用纵向卡片堆叠，每张卡片包含缩略图、名称、状态。
- `底部操作区`
  - 已解锁主题显示“使用该主题”或“当前使用中”。
  - 未解锁主题显示解锁条件，不显示购买按钮。

### 12.3 布局建议

- 页面上半部分偏“展示”，下半部分偏“选择”，两者要明显分区。
- 当前选中主题卡片可略微放大或提高亮度。
- 未解锁主题用柔和遮罩，不要完全灰死，避免失去收藏欲。

### 12.4 信息表达

- 默认主题、已解锁主题、当前使用主题、未解锁主题要四态分明。
- 解锁条件文案不宜过长，优先做成短句标签，如“累计完成 5 次每日挑战”。

## 13. 排行榜页

### 13.1 页面定位

- 排行榜是偏“成就展示”的功能页，不应做成强竞技压迫样式。
- 页面重点是清晰、可信、好切换。

### 13.2 页面总体结构

- `顶部栏`
  - 左侧返回。
  - 中间标题“排行榜”。
- `页签区`
  - 世界榜、好友榜两个页签。
  - 默认选中世界榜。
- `榜单主卡区`
  - 中央大卡片承载榜单内容。
  - 卡片宽度建议接近满宽，但左右保留明显边距。
- `底部说明区`
  - 展示规则提示、网络异常提示或好友榜隐私说明。

### 13.3 世界榜布局

- 榜单卡片内采用纵向列表。
- 每行包含：排名、头像、昵称、最高关卡、达成时间。
- 前三名可使用特殊底色或奖牌图标，但不要占据过多空间。
- 达成时间属于次级信息，字号应小于昵称和关卡。

### 13.4 好友榜布局

- 首次切换前弹出隐私协议确认弹窗。
- 同意后，主域页面仍保留外框和页签，列表内容区域交给开放数据域绘制。
- 开放数据域展示区域应固定尺寸、固定位置，避免滚动与遮罩冲突。
- 内容区外保留标题和说明，让玩家知道这里展示的是好友关系链排名。

### 13.5 设计重点

- 空榜时要有明确占位图和文案，不要留空白大框。
- 排名数字列宽固定，保证列表对齐。
- 页签切换反馈要轻快，避免像全页跳转一样生硬。

## 14. 局内页面

### 14.1 页面定位

- 局内是信息密度最高的页面，但依然要坚持“棋盘优先”。
- 所有 HUD 元素都必须为棋盘让路，中心区域是绝对核心。

### 14.2 页面总体结构

- `背景层`
  - 使用当前主题背景图。
  - 视觉焦点避开棋盘区域，避免图案太花影响识别。
- `顶部 HUD 区`
  - 放菜单按钮、机制提示图标、关卡标题、进度条、宝箱。
  - 菜单按钮位于左上安全区内。
- `中部棋盘区`
  - 页面最大主体，必须水平居中。
  - 棋盘区域周围预留消除路径特效与重力位移动画空间。
- `角色反馈区`
  - 卡皮巴拉 Combo 演出建议贴近棋盘上沿或侧边，不与顶栏抢位置。
- `底部工具区`
  - 三个工具位横向排列，位于底部安全线上方。
  - 工具位上方不要再叠加复杂信息，保证点击准确。

### 14.2.1 各主题背景图设计

- 局内主题背景建议统一使用“上轻下稳、中间净空”的构图模板。
  - 顶部放天空、树冠、灯串、蒸汽、光斑等弱信息元素，兼顾长屏延展裁切。
  - 左右两侧承担主题识别元素，形成边缘叙事，不侵入棋盘核心视区。
  - 棋盘后方中轴区域保持低对比、低纹理、低饱和，优先保证读盘和连线识别。
  - 底部使用草地、地板、桌面、石板或舞台底座承托工具区，让页面下半部更稳。
- 五个主题应共用同一套局内布局骨架，差异主要来自背景色调、边缘装饰、远景内容和局部光效，而不是改动棋盘与 HUD 位置。

- 水果主题
  - 场景关键词：晴天果园、野餐桌、果汁汽泡、夏日微风。
  - 构图建议：顶部使用浅蓝天空、软云和垂下的枝叶；左右边缘可放苹果树枝、橘子树枝与半透明果片；底部用浅木桌面或格纹野餐垫承托工具区。
  - 色彩建议：主色为蜜桃橙、草莓红、青柠绿、奶油白，整体明快清爽，但饱和度控制在中等，不可压过图块。
  - 细节元素：可点缀果汁气泡、切片水果、玻璃罐、藤篮，但不要让完整大水果贴近棋盘边缘。
  - 氛围目标：默认主题需要成为整体可读性最高的一张背景图，是局内视觉基线。

- 情绪主题
  - 场景关键词：水豚情绪小剧场、便利贴日记、梦境角落、轻喜剧治愈。
  - 构图建议：采用柔和的室内角落或梦境舞台感场景；顶部可放渐变天幕或布帘；左右边缘用低对比表情贴纸、便利贴、手绘小图标做装饰；底部用抱枕、毛毯或小舞台地板承托工具区。
  - 色彩建议：以雾蓝、淡紫、奶灰粉、暖米色为主，局部允许用少量橘红表现“冒火”“崩溃”等情绪点，但只能做边角强调。
  - 细节元素：融化表情、流泪云、黑眼圈咖啡杯、惊讶星星等更适合做贴纸式漂浮装饰，像背景情绪墙，不可成为主视觉。
  - 氛围目标：表达“内心很多戏，但外部世界依然温柔接住它”的感觉，避免做成阴郁或压抑画面。

- 森林音乐会主题
  - 场景关键词：午后林间舞台、木质演奏台、音符飘散、朋友聚会。
  - 构图建议：顶部为树冠透光与悬挂小灯串；左右可放木琴、铃鼓、三角铁、站在枝头的小鸟剪影；底部做成木质小舞台或草地舞台边缘，承接工具区。
  - 色彩建议：以木色、苔绿、浅金、湖蓝为主，整体偏暖，像阳光穿过树叶形成的轻斑驳感。
  - 细节元素：音符、五线谱弧线、飘动彩带可沿棋盘外围绕边布置，不得穿过棋盘后方中心净空区。
  - 氛围目标：传达轻快、有节奏、有人陪伴的局内氛围，但不要做出舞台追光般的强炫光。

- 植物主题
  - 场景关键词：呼吸森林、温室角落、氧气感、露珠与叶影。
  - 构图建议：顶部可用大面积树叶和透光天窗；左右两边布置龟背竹、蕨类、蘑菇和垂落藤蔓；底部用湿润石板路、苔藓地面或木质花架承托工具区。
  - 色彩建议：以森林绿 `#3D9B6E` 为主，搭配浅青绿、露珠白和少量花朵粉紫点缀，整体明度保持中高，避免变成厚重暗林。
  - 细节元素：叶片投影、漂浮花粉、露珠高光、小花开合都适合放在边缘活化画面，但笔触要柔和，像空气在呼吸。
  - 氛围目标：这是最偏沉浸治愈的一张背景，重点不是热闹，而是安静、清新、让人想深呼吸。

- 动物主题
  - 场景关键词：动物朋友聚会、河岸晒太阳、轻喜剧社交场景。
  - 构图建议：顶部为开阔天空和树枝；左右安排不同动物朋友在远景或边缘探头，如小鸟停枝、乌龟趴石、猴子挂树、鳄鱼在远处河边打盹；底部用浅草坡、河岸石块或木栈道承托工具区。
  - 色彩建议：以草绿、天蓝、暖黄、浅棕为主，整体比水果主题更自然，比植物主题更活泼。
  - 细节元素：动物必须做“陪伴式边缘出现”，不能整群挤进棋盘周围；可加入脚印、羽毛、小水波等轻量装饰增强故事感。
  - 氛围目标：突出“水豚和万物交朋友”的热闹陪伴感，但热闹只存在于边缘，中心依然让位给棋盘。

### 14.3 顶部 HUD 布局

- 左侧：关卡标题。
- 中部：进度条 + 宝箱动效。
- 右侧：机制提示图标、菜单按钮。
- HUD 采用横向一字带布局，但模块之间要通过小卡片或弱分组区分。
- 进度条应是顶部最醒目的功能组件，宝箱作为进度节点视觉焦点。

### 14.4 棋盘区布局

- 棋盘必须严格居中，周边留足动效安全边界。
- 棋盘与顶部 HUD 之间保留清晰间距，避免顶部压迫局面。
- 当盘面尺寸扩展到大规格时，优先压缩上下留白，不压缩图块点击面积。

### 14.5 底部工具区布局

- 三个工具位等距分布。
- 每个工具位包含：图标、库存数、点击反馈态。
- 缺货状态下工具位整体变灰，但仍允许点击以弹出补给弹窗。
- 工具位建议放在统一底板上，避免浮在背景里显得散。

### 14.6 设计重点

- 棋盘永远是页面视觉第一焦点。
- 顶部 HUD 尽量扁平化、横向化，减少纵向空间占用。
- 底部工具位要足够稳，不因机型变化贴到手势区。

## 15. 机制说明蒙层

### 15.1 页面定位

- 这是局内开局后的教学型短暂覆盖层，目的是快速说明特殊机制。

### 15.2 布局结构

- `半透明蒙层`
  - 仅压暗棋盘区域及周边，不完全遮死背景。
- `中心说明卡`
  - 居中展示机制图标、机制标题、简短解释。
- `辅助提示`
  - 底部可放“点击任意位置继续”或自动倒计时提示。

### 15.3 内容规范

- 每次只讲当前关卡真正涉及的机制。
- 文案控制在一到两句话内。
- 图标优先于长文，帮助玩家扫一眼就理解。

## 16. 局内菜单设置弹窗

### 16.1 页面定位

- 局内暂停入口，兼顾设置项与“重玩本关 / 放弃挑战”操作。

### 16.2 布局结构

- `标题区`
  - 中间标题“菜单”或“设置”。
- `设置项区`
  - 音效、震动、音乐三个开关项。
- `动作按钮区`
  - 主按钮：“重玩本关”。
  - 次按钮：“放弃挑战”。
  - 问题反馈按钮可置于最下方次级位置。
- `关闭区`
  - 右上角关闭按钮或底部“继续游戏”按钮二选一，建议同时保留一个更明确的“继续游戏”主路径。

### 16.3 设计重点

- 局内菜单是暂停心流的界面，应比首页设置更直接，操作按钮更突出。
- “重玩本关”和“放弃挑战”要明显区分，避免误触。

## 17. 放弃确认弹窗

### 17.1 页面定位

- 承接危险操作确认，重点是让玩家先看到“已经推进到哪里”。

### 17.2 布局结构

- `顶部情绪区`
  - 展示轻劝留插画或卡皮巴拉表情，降低生硬感。
- `当前进度区`
  - 展示当前关卡号、当前进度条、鼓励文案。
- `底部决策区`
  - 主按钮：“再试试”。
  - 次按钮：“放弃挑战并返回首页”。

### 17.3 设计重点

- 主次操作必须反转用户流失倾向，优先鼓励继续。
- “放弃挑战”按钮不要做成大面积高亮主按钮。

## 18. 道具不足弹窗

### 18.1 页面定位

- 当工具库存为 `0` 时，给出说明与补给路径。

### 18.2 布局结构

- `标题区`
  - 标题：“道具不足”。
- `道具说明区`
  - 展示当前缺少的具体道具图标、名称、功能说明。
- `次数区`
  - 显示剩余补给次数，如 `1/3`。
  - 次数应采用醒目的数字模块，而不是埋在段落里。
- `主操作区`
  - 分享补给或激励视频补给按钮。
- `辅助区`
  - 若次数为 `0`，按钮置灰并显示“今日已无补给次数”。

### 18.3 设计重点

- 道具说明要帮助玩家快速理解“为什么值得补给”。
- 主按钮文案必须明确收益，如“分享获得 1 个提示”。

## 19. 主题解锁弹窗

### 19.1 页面定位

- 这是奖励感最强的弹窗之一，需要明显区别于普通系统弹窗。

### 19.2 布局结构

- `仪式感头部`
  - 光芒、粒子、主题徽章或角色装饰形成高识别开场。
- `主题展示区`
  - 放主题名称、主题预览图、代表图块样式。
- `恭喜文案区`
  - 简短强情绪文案，如“恭喜获得 XXX 主题”。
- `按钮区`
  - 主按钮：“立即使用”。
  - 次按钮：“好的”。

### 19.3 设计重点

- 视觉中心是主题本身，而不是按钮。
- 按钮位置保持稳定，但整体版式可以比普通弹窗更高、更有庆典感。

## 20. 每日挑战完成提示

### 20.1 页面定位

- 每日挑战结束后回到首页前后的奖励确认提示。

### 20.2 布局结构

- `标题区`
  - 标题：“今日挑战完成”。
- `奖励区`
  - 突出 `50 金币`。
- `成就区`
  - 可附带显示“今日已完成”状态标签，避免用户重复进入产生疑问。
- `按钮区`
  - 单按钮“知道了”即可。

### 20.3 设计重点

- 该提示不宜过重，节奏上应比主题解锁弹窗更轻。
- 如果同时触发主题解锁，应先完成返回流程，再以主题解锁弹窗接续展示。

## 21. 通用弹窗规范

### 21.1 容器结构

- 弹窗统一由 `蒙层 + 主卡片 + 标题区 + 内容区 + 按钮区` 组成。
- 所有弹窗内部子元素必须相对弹窗自身定位，不能直接使用全屏坐标。

### 21.2 尺寸建议

- 小弹窗：适合确认类、说明类，宽度约为屏宽 `72%~80%`。
- 中弹窗：适合设置、道具不足、奖励类，宽度约为屏宽 `80%~86%`。
- 大弹窗：适合排行榜、商店这类功能页，不建议作为传统弹窗实现。

### 21.3 按钮规范

- 双按钮布局默认左次右主，或上次下主，需在全项目中保持一致。
- 危险操作一律做成次级按钮，不使用最高饱和度主按钮色。
- 单按钮弹窗按钮居中，不要偏左或偏右。

### 21.4 文案规范

- 标题 4~8 字为宜。
- 正文优先短句，不写成长段说明。
- 关键数值前置展示，减少用户阅读成本。

## 22. 页面适配与实现备注

### 22.1 全屏页面

- 首页、加载页、功能页背景都需要支持超长屏裁切。
- 重要信息尽量落在中上和中下安全区域内，不贴边。
- 设计稿中的布局参考可以按 `750 x 1334` 放置结构，但背景素材本身应按更高尺寸准备。
- 推荐做法是“固定宽度、顶部和底部留延展带”：
  - 基础出图建议：`850 x 1700`
  - 更稳妥建议：`900 x 1800`
- 美术出图时应把背景分成三段理解：
  - 顶部延展带：负责适配刘海屏、超长屏上方额外露出的区域
  - 中部主体区：角色、建筑、主景、标题承托区域，尽量稳定不被裁掉
  - 底部延展带：负责适配底部手势区以上的额外显示区域
- 如果背景只做成 `750 x 1334`，在部分长屏设备上即使布局本身正确，背景上下仍可能露出纯色兜底，因此不能把 `1334` 当作最终素材高度。

### 22.2 局内页面

- 棋盘区域优先固定结构，HUD 与工具位围绕棋盘适配。
- 若遇到小屏设备，应优先压缩背景装饰与留白，不能先压缩图块可点面积。

### 22.3 微信小游戏特殊点

- 顶部右侧要避让胶囊按钮。
- 底部按钮要避让全面屏返回横条。
- 微信原生能力按钮所在区域要提前预留，不和 PIXI 元素重叠抢点。
- 排行榜好友榜使用开放数据域时，主域需提供稳定且固定的承载窗口。

## 23. 视觉一致性建议

- 按钮统一采用圆角厚实、轻投影、按压缩放反馈。
- 货币、血量、奖励等正向信息统一使用“图标 + 数字 + 发光小动效”语义。
- 页面卡片统一使用浅色描边或柔光边框，形成系列感。
- 所有页面中的关闭、返回、菜单按钮造型统一，避免每页都长得不一样。
- 首页、商店、主题、排行榜虽然功能不同，但顶部栏结构建议保持一致，降低认知切换成本。

## 24. 页面优先级结论

- 首页要先抓“进入感”和“主按钮”。
- 局内页要先抓“棋盘识别”和“顶部进度信息”。
- 商店、主题、排行榜要先抓“顶部栏 + 页签/列表秩序”。
- 所有弹窗要先抓“主操作明确、关闭路径明确、奖励信息突出”。

以上布局设计可直接作为后续 UI 拆分、设计图细化和 PIXI 页面搭建的结构依据。
